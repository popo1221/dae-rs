## 口语化讲解区（外婆版）

### 🌐 什么是 TUN？先从"网络快递"说起

想象你寄一封信（一个数据包）：

- **物理层** = 邮局用什么车（飞机/卡车）
- **IP 层** = 信封上写的地址（从哪里来，到哪里去）——这是 TUN 工作的地方
- **TCP/UDP 层** = 信封里的内容（是挂号信还是普通信，端口号就是不同的窗口）
- **应用层** = 信的具体内容（网页/邮件）

TUN 设备就是**在信封（IP）这个层面拦截和查看**所有信件的安检口——比快递员（TCP/UDP）更底层，能看到所有信件的出发地和目的地。

### 🕵️ DNS 劫持——假传话的的艺术

**什么是 DNS？** DNS 就是"网络电话本"——你输入 `google.com`，DNS 告诉你 `142.250.x.x` 这个 IP。

**DNS 劫持是什么？** 想象你要打电话给老王，电话本告诉你的号码是错的——你打的根本不是老王。这就是 DNS 劫持：把 DNS 响应里的 IP 偷偷换成别的。

**dae-rs 为什么要劫持 DNS？**

因为很多网站根据域名决定怎么路由。如果不分青红皂白把 DNS 查询也发到代理，代理看到的只是 IP，不知道你要访问什么网站，就没法根据域名规则路由了。

所以 dae-rs 抢在 DNS 查询发出去之前，先用假的 DNS 响应"骗"客户端——返回一个**代理服务器能识别的 IP**，这样客户端后续的所有连接都会被规则引擎看到域名，路由就精准了。

**DNS 劫持的具体流程：**
1. 客户端要查 `example.com` 的 IP（发往 8.8.8.8）
2. TUN 设备截住这个包，发现是 DNS（UDP 端口 53）
3. dae-rs 自己扮演 DNS 服务器，返回 `10.0.0.1`（TUN 设备自己的 IP）
4. 客户端以为 `example.com` 对应 `10.0.0.1`，后续发往 `10.0.0.1:443` 的连接就被 dae-rs 截住了
5. 此时 dae-rs 知道客户端要访问 `example.com`，规则引擎可以精准判断：这个域名该走代理还是直连

### 📞 TCP 三次握手——打电话前互相确认

TCP 连接建立需要三步：
1. 客户端：喂，你在线吗？（SYN）
2. 服务器：在的，我也在！（SYN-ACK）
3. 客户端：好的，开始聊！（ACK）

dae-rs 的 `TcpTunSession` 就是追踪这个握手过程的——记录 SYN 发了没、SYN-ACK 收到了没、ACK 发了没。这是透明代理的"现场取证"：如果客户端说"我要发 SYN"，da e-rs 需要知道服务器回没回，没回就要重传。

### 🔀 路由决策——红绿灯指挥交通

每个数据包到了 TUN 设备，都要问规则引擎三个问题：
1. **这个域名/ IP 该走哪条路？**（代理 / 直连 / 丢弃）
2. **规则引擎怎么知道？** → 根据配置文件里的规则（domain 规则、IP 规则、GeoIP 规则等）
3. **路由完了怎么办？**
   - **代理** → 走协议处理器（VLESS/VMess/Trojan 等）
   - **直连** → 直接发到目标服务器
   - **丢弃** → 直接扔掉

### 🏠 TunProxy 的家在哪里？

TunProxy 是 dae-rs 在用户态的"前哨站"：
- **eBPF** 在内核层高速过滤流量
- **TUN** 在用户态低速但精细处理（DNS 劫持、域名规则）
- **两者配合**：内核快速判断，TUN 精准路由

## 专业结构区（同行版）

### 宏观执行流程

```
TUN 设备（物理网卡）
    ↓ 读取原始 IP 数据包
TunPacket::parse() → 解析 IP/TCP/UDP/DNS 头部
    ↓
TunProxy::should_handle() → 判断是否归 dae-rs 处理
    ↓
TunProxy::route_packet() → 规则引擎决策（Proxy / Direct / Drop）
    ↓
    ├─→ proxy_packet() → 走协议处理器（VLESS/VMess/Trojan...）
    ├─→ direct_packet() → 直连发回网络
    └─→ DNS hijack → DnsHijacker 处理，伪造响应
```

### 关键数据结构

| 结构体 | 职责 |
|--------|------|
| `TunConfig` | TUN 设备配置（IP/端口/MTU/超时） |
| `IpHeader` | 解析 IPv4 头（版本/长度/协议/源目 IP） |
| `TcpHeader` | 解析 TCP 头（SYN/ACK/FIN 标志/端口） |
| `UdpHeader` | 解析 UDP 头（端口/长度） |
| `DnsQuery` | 解析 DNS 查询（域名/QTYPE） |
| `TunPacket` | 完整数据包：raw bytes + 解析后的各层头部 |
| `DnsHijackEntry` | DNS 缓存条目：域名 → IP，带 5 分钟 TTL |
| `TcpTunSession` | TCP 会话状态追踪（SYN/Established/FinWait） |
| `UdpSessionData` | UDP 会话：client/server 两端 socket |
| `TunProxy` | 主体：持有所有状态 + 路由逻辑 |

### 核心路由决策

```
should_handle() 判断逻辑：
1. 目标是本机 TUN IP → 处理
2. 是 DNS 查询 且 目标是 dns_hijack 列表中的 IP → 处理
3. 是 outbound 方向（从本地往外发）→ 处理
4. 其他 → 不处理（可能是其他程序的网络包）
```

```
route_packet() 决策树：
RuleAction::Proxy | Default → proxy_packet()
RuleAction::Pass | Direct | MustDirect → direct_packet()
RuleAction::Drop → RouteResult::Dropped
```

### DNS Hijacker 详解

DNS Hijacker 是这个文件最复杂的子系统，工作在 `handle_query()`：

1. **查缓存**：`lookup(domain)` → 命中则直接伪造响应
2. **未命中**：转发到真实上游 DNS（默认 8.8.8.8）
3. **收到响应**：解析 IP，存入缓存（TTL 5 分钟）
4. **伪造响应**：构造 DNS 响应包，把目标 IP 换成 TUN 设备 IP

**缓存过期策略**：5 分钟 TTL（`is_expired()`），防止 DNS 记录过时。

**为什么不直接从 DNS 响应里劫持，而是要新建 UDP socket？**

因为原 UDP socket 是客户端 → DNS 服务器的，如果 dae-rs 直接修改并回复，源地址不匹配（客户端会认为不是 DNS 服务器回复的）。所以 dae-rs 必须用新 socket 发送伪造的 DNS 响应。

### IP 协议号速查

| 值 | 协议 |
|----|------|
| 1 | ICMP |
| 6 | TCP |
| 17 | UDP |
| 58 | ICMPv6 |

### TCP Flags（bitflags）

```
0x01 FIN - 结束连接
0x02 SYN - 发起连接
0x04 RST - 重置连接
0x08 PSH - 推送数据
0x10 ACK - 确认
```

## 设计取舍说明

### 1. 为什么在用户态做 TUN 而不是纯 eBPF？

**eBPF 优势**：内核层，速度极快，可处理海量流量
**eBPF 劣势**：不能解析 DNS 域名（看不到域名，只看到 IP），规则精细度受限

**dae-rs 的选择**：eBPF 快速粗筛 + TUN 用户态精细路由

- eBPF 判断"这个包要不要拦截"（4-tuple 连接级别）
- TUN 判断"拦截后走代理还是直连"（域名/IP/GeoIP 等精细规则）
- DNS 劫持必须用户态做——需要修改 DNS 响应内容

### 2. 为什么 DNS 劫持返回 TUN 设备 IP 而不是代理 IP？

因为 TUN 设备是 dae-rs 在本机创建的虚拟网卡，客户端会把"代理服务器的 IP"当作 `example.com` 的 IP，然后尝试连接它。如果连接对象是 TUN 设备自己的 IP，da e-rs 就能在同一个进程内拦截到这条连接。

### 3. DNS 缓存 TTL 为什么是 5 分钟？

太短：频繁向上游查询，增加延迟和上游压力
太长：域名 IP 变化后，缓存失效慢，可能代理到错误的 IP
5 分钟是一个经验值，平衡了缓存命中率和 DNS 记录时效性。

### 4. 为什么 TCP/UDP 会话要单独追踪？

TUN 设备的"截获"本质上是"中间人"——客户端以为在和 `example.com` 通信，实际在和 dae-rs 通信。dae-rs 必须：
- 维护每个连接的状态（SYN/Established/FinWait）
- 在连接关闭后及时清理资源
- TCP 会话追踪用于：确认三次握手完成、检测连接是否异常断开

### 5. 为什么 IPv6 支持是半残的？

代码里大量 `IpAddr::V4` 的匹配和 `u32::from(ip)` 的转换，IPv6 的 `IpAddr::V6` 分支直接返回 `0`。这是当前实现的局限——IPv6 完全支持需要更多工程投入。

### 6. 为什么 DNS 解析用 UDP 而不是 TCP？

DNS 查询通常用 UDP（端口 53），只有响应超过 512 字节才用 TCP。dae-rs 简化实现，只处理 UDP DNS 查询，这是合理的工程取舍——大多数 DNS 查询都是 UDP。

### 7. 为什么用 `RwLock` 而不是 `Mutex`？

因为读写场景：
- `tcp_sessions` / `udp_sessions`：多个并发读取 + 偶尔写入/删除
- `RwLock` 允许多个读锁并发，只有写锁才互斥
- DNS `cache` 同理

## 副作用 & 注意事项

| 副作用 | 说明 |
|--------|------|
| DNS 缓存 5 分钟过期 | 域名 IP 变更后需要等 TTL 才能生效 |
| TCP 会话超时 60s | 60 秒无活动自动清理 |
| UDP 会话超时 30s | 比 TCP 更短，因为 UDP 无连接状态 |
| IPv6 只解析不路由 | `IpAddr::V6` 被当作 `0`，会导致 IPv6 连接路由失败 |
| DNS 仅支持 UDP | 超过 512 字节的 DNS 响应（用 TCP）不会被劫持 |
| TUN MTU 1500 | 如果网络 MTU 更小，可能分包；更大则可能丢包 |

## 大白话总结（外婆版）

> tun.rs 就是 dae-rs 的"网络安检口"——在你的电脑和互联网之间建了一个虚拟邮局。
>
> 所有进出的信件（IP 包）都要先经过这个安检口。安检口有三大功能：
>
> 1. **看地址**：这封信从哪来、要到哪去（源/目 IP）
> 2. **看内容**：如果是 DNS 查询（问"这个网站在哪"），就偷偷截下来，自己去查，然后把假的地址告诉客户端——这样客户端以为在跟某个 IP 通信，实际在跟 dae-rs 通信，dae-rs 就能决定要不要帮它代理
> 3. **指路**：根据规则引擎判断，这个连接应该直连还是走代理，还是直接扔掉
>
> 整个系统靠 DNS 劫持这个"小聪明"实现了精准的域名级路由——不用劫持 DNS 的话，代理只能看到 IP，看不到域名，就没法用 `domain:` 规则了。
