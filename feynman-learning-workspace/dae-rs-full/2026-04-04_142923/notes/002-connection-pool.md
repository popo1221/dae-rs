## 口语化讲解区（外婆版）

**为什么需要连接池？**

想象你要寄信：
- **不用连接池**：每寄一封信都要去邮局开户、寄完销户——每次都要三次握手建立连接、四次挥手断开，效率很低
- **用连接池**：开了户之后连接一直保持，下次寄信直接用这个连接，不用重新开户

这就是 TCP keepalive 的原理——连接池让连接被复用，不每次重建。

**`get_or_create()` 双检锁的原理（重点）**：

去柜子里拿玩具：
1. **第一遍**（读锁）：往柜子里瞄一眼，有就直接拿（快速路径）
2. **第二遍**（写锁）：如果瞄不到（没建立连接），拿到写锁再瞄一遍（防止两个小朋友同时瞄到空的、重复建连接），确认真的没有，建一个

为什么第二遍还要检查？因为两个小朋友可能同时来，都瞄到空的，都去拿写锁——写锁是排他的，第二个拿到写锁的人进去一看，发现第一个已经建好了，就直接用。

**IPv6 降级问题（踩坑点）**：

如果地址转换失败，这个柜子会悄悄把 IPv6 地址变成 `0.0.0.0:0`，相当于把所有 IPv6 连接都变成"无名氏"，会导致连接复用失效。这个降级策略是"不崩溃但可能有问题"——好过程序直接崩掉。

## 专业结构区（同行版）

### 函数签名

#### 输入参数

| 参数名 | 数据类型 | 业务含义 | 可选？默认值 |
|--------|---------|---------|------------|
| `src: SocketAddr` | SocketAddr | 源地址（客户端地址） | 必填 |
| `dst: SocketAddr` | SocketAddr | 目标地址（服务器地址） | 必填 |
| `proto: Protocol` | enum | TCP 还是 UDP | 必填 |

#### 返回值

| 情况 | 返回类型 | 业务含义 |
|------|---------|---------|
| 正常（缓存命中） | `(SharedConnection, false)` | 复用已有连接，"false"=没新建 |
| 正常（缓存未命中） | `(SharedConnection, true)` | 新建了连接，"true"=新建 |
| IPv6 转换失败 | 静默降级到 `0.0.0.0:0` | 警告日志，但不断言 |

#### 副作用

- `get_or_create` 会插入 HashMap（缓存未命中时）
- `cleanup_expired` 会删除 HashMap 条目
- `close_all` 会触发所有连接的优雅关闭

---

### 宏观执行流程

**连接获取（`get_or_create`）**：
1. 先拿读锁，查 HashMap 是否有缓存 → 有则复用（fast path）
2. 缓存未命中，拿写锁
3. 双检：写锁拿到后再查一遍 HashMap（防止并发重复创建）
4. 用 `new_connection()` 建立新连接
5. 插入 HashMap，返回

**连接清理（`cleanup_expired`）**：
1. 每 10 秒被 proxy 的定时任务调用一次
2. 遍历 HashMap，对每条连接判断是否超时
3. TCP 用 `connection_timeout`（默认 60s）
4. UDP 用 `udp_timeout`（默认 30s，更短）
5. `retain()` 删除超时的条目

**全局关闭（`close_all`）**：
1. `connections.drain()` 把 HashMap 里所有条目一次性清空
2. 对每条连接调用 `start_close()`

---

### 关键逻辑块

- **双检锁（double-checked locking）**：`get_or_create` 的核心——读锁 fast path + 写锁 slow path，既保证并发安全又减少锁竞争
- **双版本号存储 CompactIp**：`version(4bits) | address(124bits)` 合并到一个 `u128` 里，IPv4 和 IPv6 共用同一个结构做 HashMap key
- **protocol 字段**：`6=TCP`，`17=UDP`（标准 IANA 协议号）
- **`drain()` vs `clear()`**：drain 后 HashMap 变为空但保留容量（不会重新分配内存），性能更好

---

### 核心数据结构

| 结构 | 字段 | 作用 | 类比 |
|------|------|------|------|
| `CompactIp` | `u128` | 压缩存储 IPv4/IPv6，用版本号区分 | 把邮政编码和地址编码成一维条形码 |
| `ConnectionKey` | 5-tuple | 唯一确定一条连接 | 信封上的"寄件人+收件人+协议" |
| `ConnectionPool` | `HashMap<ConnectionKey, SharedConnection>` | 连接缓存 | 玩具共享柜 |
| `SharedConnection` | `Arc<RwLock<Connection>>` | 允许多个读取者、一个写入者并发访问 | 柜子里每件玩具都有锁 |

### 设计意图

- **为什么用 CompactIp 而不用 `(IpAddr, Port)` 直接做 key？** — `IpAddr` 是 enum（`V4(Ipv4Addr)` 或 `V6(Ipv6Addr)`），无法直接 `Hash`，需要 wrap 成固定大小的 `u128` 才能放进 HashMap
- **为什么 UDP 超时比 TCP 短（30s vs 60s）？** — UDP 是无连接协议，session 靠应用层维护，超时设短避免无效会话占用内存
- **为什么 cleanup 用 `retain` 而不是逐个 remove？** — `retain` 只遍历一次 O(n)，逐个 remove 每次都要再查找 O(n²)
- **为什么用 `Arc<RwLock<Connection>>` 而不是 `Arc<Mutex<Connection>>`？** — 读多写少：连接大部分时候只是被读取状态（`is_expired` 检查），RwLock 允许多读并发，Mutex 只能单读

---

### 边界与异常

- **IPv6 静默降级**：如果 `to_socket_addrs()` 失败，源/目标地址会变成 `0.0.0.0:0`，导致这些连接永远无法复用（bug 风险，注释已标注）
- **连接池爆满**：HashMap 无上限，如果连接建了不清理，会耗尽内存（通过 `cleanup_expired` 缓解）
- **double-check 并发**：两个协程同时发现缓存未命中，第二个拿到写锁的会覆盖第一个的插入（同一个 key，value 相同，不影响）

## 关键应用场景

### 🗺️ 连接复用——TCP 代理的性能核心

**这段代码在系统中做什么**：
`connection_pool.rs` 让 TCP/UDP 连接被复用——同一个"寄件人→收件人"通道建一次用多次，不用每次都重新建立（不用每次都三次握手）。

**调用链追溯**：
```
TcpProxy::start()
  → 每个新连接进来
  → connection_pool.get_or_create(key)  // key = 5-tuple
    → HashMap 里找，有则复用，无则新建
  → 复用已有连接 → 减少三次握手
  → proxy 每10秒 cleanup_expired() // 清理超时连接
```

**触发的关键决策**：

| 场景 | 输入 | 决策 | 后果 |
|------|------|------|------|
| 第一个连接 | 新建 key | `get_or_create(key)` → 新建 | 三次握手，耗时 50-100ms |
| 第二个相同连接 | 相同 key | 直接取缓存 | 0ms，直接复用 |
| 超时没人用 | 60s 无活动 | `cleanup_expired()` 删除 | 释放内存 |

**如果这段代码消失会怎样**：
每次请求都重新建立 TCP 连接——三次握手 + TLS 握手耗时变成 100-300ms/请求，网络吞吐量暴跌到原来 1% 以下。连接复用是代理性能的关键。

**和同类方案相比为什么选它不选别的**：
- **不用连接池 vs 用连接池**：无池 = 每次新建（慢）；有池 = 复用（快）
- **用 Mutex vs RwLock**：Mutex 只允许单读，RwLock 允许多读并发（连接大部分时间只读状态）
- **`HashMap::new()` vs `drain()`**：drain 清空但保留容量，避免重新分配

## 大白话总结（外婆版）

> 连接池就像游乐场的"快速通行证"。
>
> 小朋友第一次来玩，要先去前台登记（TCP 三次握手），领一张卡。以后每次来，直接刷卡就行，不用重新登记。通行证有效期到了（超时），前台会自动把它收回（cleanup）。
>
> 这个柜子（`HashMap`）放的不是玩具，是一张张"连接卡"。每次有小朋友来（新的 TCP 连接），先去柜子里找找有没有他的卡；有就直接用，没有就去前台办一张，下次就能用了。
>
> 关键设计：柜子上了一把"双检锁"——先隔着玻璃门看一眼有没有（读锁，快），没有再开锁进去找（写锁，慢）。这样大多数时候不用开锁，只有真的没有的时候才开，效率最高。
