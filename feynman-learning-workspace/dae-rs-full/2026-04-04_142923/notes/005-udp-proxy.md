## 口语化讲解区（外婆版）

**TCP vs UDP 的本质区别**

- **TCP 是打电话**：先建立连接（三次握手），通话过程中双方一直保持通话，直到一方挂断（四次挥手）
- **UDP 是发快递**：没有"建立连接"这回事，每个快递（数据包）都是独立的，寄出去就不管了，不知道对方收没收到

所以 TCP 叫"有连接"，UDP 叫"无连接"。

**为什么 UDP 也要"会话"管理？**

UDP 虽然不建立连接，但实际使用中还是有"会话"的概念——比如 DNS 查询，我发一个问题过去（DNS 服务器），期望收到一个答案。如果每个包都随机选出口，就收不到对应的回答了。

所以要管理一个"会话表"：寄件人地址 → 快递柜格子（socket）。

**`session_relay_loop` 是怎么工作的？**

```
快递柜里有两格：
格子A：通往客户端
格子B：通往DNS服务器

loop:
  随时监听 A ← B 两边的动静
  A 有快递来 → 往 B 寄
  B 有快递来 → 往 A 寄
  如果超过30秒没有任何动静 → 关掉这个快递柜
```

**`tokio::select!` 的多路复用**：

`tokio::select!` 就像快递柜同时盯着两个方向（A 和 B），哪个方向有动静就先处理哪个。如果两边同时来了两个快递，先处理先到的那个，另一个等着。

## 专业结构区（同行版）

### 宏观执行流程

**收到 UDP 包（`handle_packet()`）**：
1. 从包数据提取目标地址
2. 用 5-tuple 查 sessions HashMap
3. 命中 → 直接转发到已有 session
4. 未命中 → `create_session()` 创建新会话

**创建会话（`create_session()`）**：
1. 创建 socket 对（一对 UDP socket）
2. 存入 sessions HashMap
3. relay 第一个包
4. spawn `session_relay_loop()` 后台运行

**会话循环（`session_relay_loop()`）**：
1. `tokio::select!` 多路监听 client 和 remote 两个方向
2. 有数据就转发到对面
3. 超时（per_session_timeout，默认 30s）则退出循环
4. 退出后删除 session

---

### 关键逻辑块

- **`tokio::select!` 三路分支**：`client_recv | server_recv | sleep(1s)` — 第三个分支每 1 秒触发一次，用于定期检查超时（因为 `select!` 不支持 `elapsed` 定时）
- **NAT socket 对**：客户端 UDP socket + 服务器 UDP socket，对外伪装成两个独立连接
- **`timeout` vs `elapsed` 判断**：`last_activity.elapsed() > timeout` 是同步判断，不能直接在 `select!` 里用，所以用每秒 tick 来兜底

---

### 设计意图

- **为什么 UDP 超时比 TCP 短（30s vs 60s）？** — UDP 无连接，服务器不主动关闭连接，session 全靠客户端活动驱动；如果客户端停了，UDP session 没有意义，应该快速清理释放资源
- **为什么用 HashMap 而不是 Vec？** — 用 5-tuple（ConnectionKey）做 key 查询 O(1)，如果用 Vec 要 O(n) 查找，高并发下性能差
- **`extract_destination` 是简化实现**：当前代码返回硬编码 `8.8.8.8:53`，实际应该解析 SOCKS5/HTTP/DNS 等协议提取真实目标

## 关键应用场景

**系统中做什么**：
UDP 代理管理"伪连接"（NAT session）——每个 UDP 会话创建一个 socket 对，客户端和远程之间双向转发数据包，通过超时自动清理。

**没了会怎样**：
UDP 流量无法被代理，所有 DNS-over-UDP、QUIC 等基于 UDP 的协议无法工作。

## 大白话总结（外婆版）

> udp.rs 就是"快递柜管理员"。
>
> TCP 是打电话，必须先接通才能说话（建立连接）。UDP 不建立连接，直接发快递，每个快递都是独立的。
>
> 但问题是：如果一次 DNS 查询，发了问题过去，却收到另一个 DNS 服务器的回答，就乱套了。
>
> 所以这个文件建了一个"快递柜"——每来一个新的 UDP 会话（快递寄件），就在柜子里分配一个格子（socket 对），记录"这个寄件人的快递以后都走这个格子"。超过 30 秒没动静，就把这个格子关掉，腾出空间给新的会话。
