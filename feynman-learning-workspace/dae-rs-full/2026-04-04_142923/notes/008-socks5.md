## 口语化讲解区（外婆版）

**SOCKS5 是怎么工作的？**

场景：你（A）想买隔壁店的奶茶，但不想让奶茶店知道是你买的（隐藏身份）。你找了一个店员（B）帮忙买。

SOCKS5 就是规定 A 和 B 怎么说话的协议：

```
第一步（握手）：A跟B说："我支持两种买法：自己买单 or 报我会员号" → B回："好，用会员号"
第二步（命令）：A跟B说："帮我去买杯奶茶，地址是XXX路YYY号" → B回："好，我帮你联系"
第三步（转发）：B去买奶茶，奶茶店只知道B，不知道A
```

这就是 SOCKS5 CONNECT 命令的三步：Greeting → Auth → Command。

**支持三种地址类型**：

| 类型 | 格式 | 例子 |
|------|------|------|
| IPv4 | 4字节 IP + 2字节端口 | `192.168.1.1:80` |
| IPv6 | 16字节 IP + 2字节端口 | `[::1]:80` |
| 域名 | 1字节长度+域名+2字节端口 | `google.com:80` |

**为什么要支持域名地址？**
因为有些代理需要直接知道域名来做路由/过滤，不能先把域名解析成 IP 再传。

## 专业结构区（同行版）

### RFC 1928 SOCKS5 握手流程

```
Client → Server: Greeting (VER=05 + AUTH_METHODS)
Server → Client: Choice (VER=05 + SELECTED_METHOD)

Client → Server: Auth (METHOD_SPECIFIC)
Server → Client: Auth Result

Client → Server: CONNECT Request (VER + CMD + ATYP + DST.ADDR + DST.PORT)
Server → Client: Reply (VER + REP + ATYP + BND.ADDR + BND.PORT)
```

### ATYP 地址格式

| ATYP | 地址类型 | 格式 |
|------|---------|------|
| 0x01 | IPv4 | `[0x01][4字节IP][2字节端口]` |
| 0x03 | 域名 | `[0x03][1字节长度][域名][2字节端口]` |
| 0x04 | IPv6 | `[0x04][16字节IP][2字节端口]` |

### REP 回复码

| 值 | 含义 |
|----|------|
| 0x00 | 成功 |
| 0x01 | 一般失败 |
| 0x02 | 规则不允许 |
| 0x03 | 网络不可达 |
| 0x04 | 主机不可达 |
| 0x05 | 连接被拒绝 |

## 关键应用场景

**系统中做什么**：
SOCKS5 是 dae-rs 入口协议之一——用户在客户端配置 SOCKS5 代理，客户端通过 SOCKS5 协议连接到 dae-rs，dae-rs 再根据规则决定直连还是走上游代理。

**没了会怎样**：
用户无法通过 SOCKS5 协议使用 dae-rs 代理功能。

## 大白话总结（外婆版）

> socks5.rs 就是"代购协议手册"。
>
> 你想买一样东西，但不想自己出面，就找个代购帮你买。代购协议手册规定了你（客户端）和代购（dae-rs）之间怎么说话：
>
> 1. **打招呼**："我有两种身份验证方式，你要用哪个？" → "用无验证的吧"
> 2. **下订单**："帮我买杯奶茶，地址是XX奶茶店YY号" → "好，马上帮你联系"
> 3. **代购完成**：代购帮你买好，还告诉你"我从XX地址帮你买的"（BND.ADDR = 代购的出口地址）
>
> 地址格式支持三种：门牌号（IPv4）、大段地址（IPv6）、还有直接写店名（域名）——这样代购可以根据店名决定去哪家分店买。
