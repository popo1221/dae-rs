# Ralph Mode: dae-rs 流量代理全过程跟踪

## 目标
在 dae-rs 中实现流量代理全过程的端到端可观测性，将每个关键阶段都接入 TrackingStore。

## 现状

dae-rs 已有 TrackingStore 系统，位于 `crates/dae-proxy/src/tracking/`：
- `store.rs` — TrackingStore 主类，包含 Connection/Node/Rule/Protocol tracking
- `types.rs` — 跟踪数据类型定义
- `constants.rs` — 枚举常量
- `maps.rs` — eBPF stats maps

已接入的模块：
- `protocol_dispatcher.rs` — 连接生命周期 (New→Established→Closed) + 字节计数
- `proxy/lifecycle.rs` — HTTP API + Prometheus 导出

## 缺口（需要实现的）

### P1 - DNS 跟踪
- `crates/dae-proxy/src/dns/mac_dns.rs` 中的 `MacDnsResolver` 未接入 TrackingStore
- 需要跟踪: DNS 查询次数、延迟、缓存命中率、upstream 切换、错误

### P1 - eBPF stats bridge
- `crates/dae-tc/`, `crates/dae-xdp/`, `crates/dae-ebpf-direct/` 的 stats 未同步到 TrackingStore
- eBPF 程序有 PerCpuArray stats 和 SESSIONS map，但用户态未读取
- 推荐方案: 在 `EbpfContext` 实现 poll_stats() 定时批量同步

### P2 - Proxy chain 多跳跟踪
- `crates/dae-proxy/src/proxy_chain.rs` 中每个 hop 未单独统计
- 需要: 每个 hop 的独立 ConnectionKey、hop_latency、hop_status

### P2 - 协议解析层跟踪
- Trojan: `crates/dae-protocol-trojan/src/protocol.rs` — 命令类型、解析耗时
- VMess: `crates/dae-protocol-vmess/` — UUID、加密方式、解析耗时
- Shadowsocks: `crates/dae-protocol-shadowsocks/` — 加密方式、解析耗时

### P2 - TLS/QUIC handshake 跟踪
- `crates/dae-proxy/src/transport/tls.rs` — TLS 握手开始/结束、协商结果
- `crates/dae-protocol-tuic/`, `crates/dae-protocol-hysteria2/` — QUIC 连接建立时间、握手 RTT

## 实现要求

### 1. DNS 跟踪实现步骤
在 `crates/dae-proxy/src/dns/mac_dns.rs` 中：
1. 给 `MacDnsResolver` 添加 `tracking_store: Option<Arc<TrackingStore>>` 字段
2. 添加 `with_tracking_store()` 构造函数
3. 在 `resolve()` 方法中:
   - 缓存命中时调用 `store.record_dns_cache_hit()`
   - 缓存 miss 发起查询时调用 `store.record_dns_cache_miss()`
   - 查询返回时记录 latency: `store.record_dns_query(latency_ms)`
   - upstream 切换时: `store.record_dns_upstream_switch()`
   - 错误时: `store.record_dns_error()`

在 `crates/dae-proxy/src/tracking/types.rs` 中 `OverallStats` 添加:
- `dns_queries_total: u64`
- `dns_cache_hits: u64`
- `dns_cache_misses: u64`
- `dns_upstream_switches: u64`
- `dns_errors: u64`
- `dns_latency_sum_ms: u64`
- `dns_latency_count: u64`

在 `crates/dae-proxy/src/tracking/store.rs` 添加对应方法。

### 2. eBPF stats bridge 实现步骤
在 `crates/dae-proxy/src/ebpf_integration/` 或 `TrackingStore` 中：
1. 添加 `poll_stats()` 方法或 StatsPoller
2. 定时读取 eBPF PerCpuArray stats，合并到 `TrackingStore.overall`
3. 读取 SESSIONS map 更新 connection tracking
4. 第一阶段: 简单轮询 (e.g., 5s interval)，合并到 overall stats

### 3. Proxy chain 多跳跟踪
在 `crates/dae-proxy/src/proxy_chain.rs` 中:
1. 为每个 hop 创建独立 `ConnectionKey`
2. 每个 hop 结束时记录: `store.update_connection(hop_key, stats)`

### 4. 协议解析层跟踪
在协议解析入口处 (Trojan/VMess/Shadowsocks handler):
1. 解析开始时记录 timestamp
2. 解析完成时计算 latency 并上报
3. 错误时记录 error

### 5. TLS/QUIC handshake 跟踪
在 TLS/QUIC transport 建立连接时:
1. `handshake_start` 事件
2. `handshake_end` 事件，记录 success/failure
3. 记录协商的 TLS 版本、cipher suite 等
